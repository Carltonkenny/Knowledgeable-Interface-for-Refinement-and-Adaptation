Forge Orchestrator — Complete Logic

Forge is not a router. It is a personality with routing capability. Every decision it makes is expressed in character. It reads the user's profile before speaking, adapts its tone, and decides in one structured LLM call whether to ask, proceed, or flag complexity.

DECISION LOGIC
msg.length < 10 chars → CONVERSATION
ambiguity_score > 0.7 → ASK 1 QUESTION
references prev output → FOLLOWUP
new domain prompt → FULL SWARM
known domain (profile) → SKIP DOMAIN AGENT
ORCHESTRATOR OUTPUT JSON
{
  "user_message": "...",
  "proceed_with_swarm": true,
  "agents_to_run": ["intent", "domain"],
  "skip_reason": "context skipped—no history",
  "clarification_needed": false,
  "clarification_question": null,
  "user_facing_message": "Running it. Treating this as B2B copy—adjust me if wrong.",
  "tone_used": "direct",
  "profile_applied": true
}
KEY DESIGN DECISION

Forge makes ONE LLM call: 150 tokens max, fast model (Gemini Flash / GPT-4o-mini), returns structured JSON. The user_facing_message is sent to the client immediately via SSE. If proceed_with_swarm: true, agents fire in parallel while the user reads Forge's message. Total overhead: ~300-500ms.

User Profile System — The Personalization Brain

The profile is never written by the user. It is generated automatically by the Profile Updater agent after each session. Forge reads it before every interaction and adapts its personality expression accordingly.

Example User Profile — "Alex" (47 sessions)

DOMAIN CONFIDENCEmarketing · 94%
PROMPT SPECIFICITY TRENDimproving
CLARIFICATION ACCEPTANCE80%
TECHNICAL LEVELmedium
PATTERNS LEARNED:
· Always omits audience from prompts
· Prefers direct tone in Forge responses
· B2B SaaS context in 80% of prompts
· Responds well to quality scores

How Profile Changes Forge Behavior

DOMAIN KNOWN → SKIP DOMAIN AGENT
Saves one LLM call. Forge injects domain context directly into prompt engineer prompt.
OMITS AUDIENCE → FORGE ASKS
Orchestrator sets clarification_needed:true, asks specifically about audience.
PREFERS DIRECT → NO WARMUP
Forge skips explanations, gets to the point immediately. Same core personality, different expression.
IMPROVING QUALITY → ACKNOWLEDGE RARELY
Forge occasionally notes improvement. Not every session — only when meaningful milestone hit.
PROFILE UPDATER AGENT — RUNS ASYNC AFTER SESSION
Input
existing profile JSON + last 5 sessions of interactions + quality scores + clarification outcomes
LLM Call
fast model · 500 tokens · returns updated profile JSON · runs in FastAPI BackgroundTask
Trigger
30 min inactivity OR explicit session end OR every 5th interaction within a session

MCP Integration — Progressive Trust Model

MCP is always available. But personalization depth scales with session count. Users feel progression naturally — they don't hit a wall.

// Debate: Hard gate vs progressive model — progressive wins. Users self-select into deeper engagement.

Level 0 · Cold
0–10
SESSIONS
MCP works. Forge uses no profile — generic personality, all agents run, no domain assumptions.
Forge tells user: "Use me via the app a few more times. I'll get sharper here."
Level 1 · Warm
10–30
SESSIONS
Profile has enough signal. Domain agent skips on known domains. Forge adapts tone. Clarification questions become targeted.
Domain skip active · Tone adaptation active
Level 2 · Tuned
30+
SESSIONS
Full profile active. Forge references your patterns, skips unnecessary analysis, adapts fully. Feels like a senior collaborator who knows your work.
Full personalization · Pattern references · History-aware
